Dosen sering menanyakan di mana letak logika bisnis, di mana letak rute, dan bagaimana kode frontend bisa menyatu dengan backend. Berikut adalah rincian fungsional struktur folder proyek:

1.1 Sisi Backend (Laravel 10)
-----------------------------
* app/Http/Controllers/Web/: Tempat menyimpan logika kontroler web. Controller bertugas mengambil data dari Model, memproses otorisasi peran pengguna (role checking), melakukan manipulasi data, dan meneruskan data tersebut ke tampilan frontend menggunakan Inertia rendering.
* app/Http/Controllers/Api/: Berisi file ApiAuthController dan ApiAsetController untuk melayani request mobile tracker Android Kotlin dalam format JSON.
* app/Models/: Berisi Model Eloquent (User.php, Aset.php, Ruangan.php, Mutasi.php, Troubleshoot.php, TroubleshootLog.php, Pemeliharaan.php, PemeliharaanLog.php, DokumenMutu.php) yang memetakan tabel database MySQL secara objek ORM. Model mendefinisikan relasi (seperti Aset milik Ruangan melalui belongsTo).
* database/migrations/: Skema pembuatan tabel database. Skema migrations dipisahkan agar pembuatan constraint foreign key dan indexing berjalan runtut tanpa error.
* database/seeders/DatabaseSeeder.php: Menyimpan data awal bawaan (akun user dengan role berbeda seperti Admin_IPSRS, Staf_Logistik, dan Unit_RS, ruangan medis, serta data aset medis dummy).
* routes/web.php: Menyimpan rute akses web client yang dilindungi oleh middleware 'auth' (mengharuskan login session). Rute mengarah ke controller Inertia.
* routes/api.php: Menyimpan rute REST API untuk Android. Rute login bersifat publik, sedangkan rute lainnya dilindungi middleware 'auth:sanctum' (wajib menyertakan token).

1.2 Sisi Frontend (React.js + Tailwind)
---------------------------------------
* resources/js/Pages/: Halaman antarmuka pengguna (Views). Setiap berkas disini (seperti Dashboard.jsx, Aset/Index.jsx, WorkOrder/Index.jsx, Mutasi/Index.jsx, Direktori/Index.jsx, Dokumen/Index.jsx) adalah komponen React utuh yang merender UI halaman web secara dinamis.
* resources/js/Layouts/AuthenticatedLayout.jsx: Kerangka halaman utama (Sidebar, Topbar/Header, Profil Info, Lonceng Notifikasi Dinamis, Kamera Web Scanner QR Code) yang membungkus semua halaman setelah pengguna berhasil login. Menghindari penulisan sidebar berulang di setiap file Page.
* resources/js/app.jsx: Entri utama JavaScript. Di sini Inertia dikonfigurasi untuk memuat komponen React dari folder Pages secara dinamis.

BAB II: ALUR DATA DAN REQUEST-RESPONSE LIFECYCLE (BAGAIMANA APLIKASI BEKERJA)

2.1 Siklus Request Web (Menggunakan Protokol Inertia.js)
--------------------------------------------------------
Inertia bertindak sebagai jembatan. Alur kerja ketika pengguna mengakses halaman Aset Medis adalah sebagai berikut:
* 1. Pengguna mengklik menu 'Aset Medis' di Sidebar.
* 2. Browser mengirimkan request internal (Inertia request XHR/AJAX) ke rute /aset.
* 3. Laravel mencocokkan rute di web.php dan memanggil AsetController@index.
* 4. AsetController melakukan query database melalui Eloquent: Aset::with('ruangan')->get() untuk memuat aset medis beserta nama ruangan tempat alat itu ditempatkan.
* 5. Controller mengembalikan respon: return Inertia::render('Aset/Index', ['asets' => $asets]);
* 6. Inertia menangkap kembalian tersebut, lalu mengirimkan payload berupa data JSON (bukan HTML mentah) ke frontend.
* 7. Komponen React di resources/js/Pages/Aset/Index.jsx menerima data tersebut melalui props (asets) lalu merendernya secara dinamis di browser tanpa refresh halaman penuh.

2.2 Siklus Request API Mobile (Laravel Sanctum)
-----------------------------------------------
Alur kerja ketika aplikasi Android pelacak lokasi meminta daftar barang atau mengirimkan koordinat GPS ke server:
* 1. Aplikasi Android mengirim HTTP POST request ke http://alamat-server/api/tracking.
* 2. Android menyertakan header Authorization: Bearer {token} yang diperoleh saat login.
* 3. Laravel mencocokkan rute di routes/api.php dan memprosesnya melalui middleware auth:sanctum.
* 4. Middleware memverifikasi keaslian token pada tabel personal_access_tokens di database.
* 5. Jika valid, request diteruskan ke Api\ApiAsetController@updateTracking yang mengambil data koordinat GPS terbaru dan menyimpannya ke tabel t_aset.

3.1 Logika Mutasi Aset & Keamanan Database Transactions
-------------------------------------------------------
Fokus Pembahasan: Pertanyaan Dosen: 'Bagaimana logika mutasi alat medis dan bagaimana Anda menjamin konsistensi lokasi?'
Di dalam file MutasiController.php, transaksi dilakukan di dalam blok DB::transaction(). Hal ini penting untuk memastikan sifat Atomicity (semua berhasil atau tidak sama sekali). Logikanya:
1. Validasi input: memastikan aset yang akan dimutasi saat ini memiliki data asal unit ruangan yang terdaftar di database.
2. Jika validasi lolos, jalankan DB::transaction(function() { ... })
3. Sistem memperbarui kolom id_ruang_saat_ini pada tabel t_aset menjadi ID ruangan tujuan mutasi.
4. Sistem mencatat log riwayat pemindahan ke dalam tabel t_mutasi.
5. Jika terjadi error di tengah proses (misalnya database down saat mencatat log mutasi), Laravel otomatis membatalkan pembaruan ruangan pada aset (Rollback) agar lokasi aset tidak bergeser secara tidak konsisten.

3.2 Logika Pelaporan Kerusakan & Penugasan Teknisi (Work Order)
---------------------------------------------------------------
Fokus Pembahasan: Pertanyaan Dosen: 'Bagaimana alur pelaporan kerusakan alat medis dan penugasan teknisi?'
Tiket troubleshoot/kerusakan dikelola melalui tabel t_troubleshoot. Kolom penting di sini adalah status (status_ticket dengan opsi Open, Pengecekan, Dikerjakan, Menunggu Sign-Off, Closed), id_user_pelapor, dan id_teknisi_penanggungjawab.
1. Pengguna dengan peran 'Unit_RS' mengirim form laporan kerusakan alat dari unitnya. Status awal diset default ke 'Open'.
2. Pengguna dengan peran 'Admin_IPSRS' masuk ke menu Work Order, melihat tiket 'Open' tersebut, lalu menugaskan teknisi penanggungjawab dengan memilih dari daftar user yang memiliki kompetensi teknis.
3. Teknisi lapangan (seperti Budi, Hendra, Agus) masuk ke sistem. Mereka hanya melihat tiket pengerjaan yang ditugaskan kepada mereka dan dapat memperbarui status pengerjaan secara bertahap (misal dari Open ke Dikerjakan).
4. Setelah selesai dikerjakan, status tiket diperbarui menjadi 'Menunggu Sign-Off' untuk divalidasi oleh admin atau unit sebelum tiket ditutup secara permanen ('Closed').

3.3 Pengelolaan Laporan Cetak PDF (Print-Friendly CSS)
------------------------------------------------------
Fokus Pembahasan: Pertanyaan Dosen: 'Bagaimana laporan dicetak menjadi PDF?'
Di halaman resources/js/Pages/Aset/Index.jsx, tombol 'Cetak Laporan' memicu fungsi window.print(). Untuk menghasilkan PDF yang rapi tanpa elemen navigasi web, kita menggunakan utility class Tailwind CSS khusus cetak:
* print:hidden -> Menyembunyikan sidebar, topbar, dan tombol cetak saat proses print PDF berlangsung.
* print:block -> Memunculkan Kop Laporan resmi yang disembunyikan di layar monitor.
* print:shadow-none & print:border-none -> Menghilangkan bayangan dan border dekoratif agar PDF bersih layaknya dokumen kertas resmi.
* print:text-black & print:divide-slate-350 -> Mengubah teks menjadi hitam pekat agar hasil cetakan jelas dibaca.

BAB IV: BEDAH DETAIL FILE KODE PROGRAM (FUNGSI DAN PENJELASAN KODE)

4.1 File Migrasi Database
-------------------------
Berkas ini berfungsi untuk mendefinisikan skema tabel database MySQL. Di dalamnya terdapat fungsi up() yang dijalankan saat php artisan migrate dipanggil. Dosen akan menyorot relasi foreign key di sini:
* Tabel Ruangan: Dibuat dengan id_ruang sebagai primary key dan nama_ruang ber-constraint unique() agar tidak ada ruangan ganda.
* Tabel Aset: Memiliki kolom id_ruang_saat_ini yang dihubungkan sebagai foreign key menggunakan constrained('m_ruangan')->nullOnDelete(). Artinya, jika sebuah ruangan dihapus, kolom lokasi aset diisi NULL tanpa menghapus data aset.
* Tabel Mutasi: Melacak log perpindahan aset (id_aset, id_ruang_asal, id_ruang_tujuan).
* Tabel Troubleshoot: Melacak tiket kerusakan (id_aset, id_user_pelapor, id_teknisi_penanggungjawab, status_ticket).

4.2 File Model (app/Models/Aset.php)
------------------------------------
Model Eloquent bertindak sebagai perwakilan tabel database di pemrograman PHP berbasis objek. Di dalamnya, relasi didefinisikan secara eksplisit untuk mempermudah query data:
* protected $fillable: Daftar kolom yang diizinkan untuk diisi secara massal (mass assignment) saat proses insert/update data di controller.
* public function ruangan(): Mendefinisikan relasi belongsTo(Ruangan::class, 'id_ruang_saat_ini'). Relasi ini memberitahu Laravel bahwa satu aset berada di satu ruangan (One-to-One/Many-to-One).
* public function troubleshoot(): Mendefinisikan relasi hasMany(Troubleshoot::class, 'id_aset'). Berarti satu aset bisa memiliki banyak catatan riwayat kerusakan/Work Order (One-to-Many).

4.3 File Controller Aset (app/Http/Controllers/Web/AsetController.php)
----------------------------------------------------------------------
Controller ini melayani rute halaman Aset web. Dosen biasanya menanyakan bagaimana validasi peran dan query databasenya bekerja di sini:
* Proteksi Peran (Role Checking): Di awal metode CRUD, terdapat pengecekan menggunakan helper checkCanCreate dan checkCanEditOrDelete untuk membatasi Unit_RS, Staf_Logistik (hanya diizinkan tambah/create, dilarang edit/hapus), serta teknisi budi_ipsrs, hendra_ipsrs, dan agus_ipsrs agar tidak bisa memanipulasi data aset master.
* Eager Loading (with): Mengambil data aset menggunakan Aset::with('ruangan')->get(). Penggunaan 'with' bertujuan mengoptimalkan database dengan teknik Eager Loading untuk mencegah masalah N+1 Query.
* Inertia::render(): Mengirimkan data $asets ke frontend React Page Aset/Index.jsx.

4.4 File Controller Mutasi (app/Http/Controllers/Web/MutasiController.php)
--------------------------------------------------------------------------
Menangani transaksi mutasi aset medis. Di dalamnya terdapat fungsi kritis DB::transaction() untuk mengubah kolom id_ruang_saat_ini di tabel t_aset dan membuat record baru di tabel t_mutasi secara transaksional.

4.5 File Controller Work Order (app/Http/Controllers/Web/WorkOrderController.php)

Menangani pembuatan tiket laporan kerusakan oleh Unit_RS, penunjukan teknisi oleh Admin IPSRS, dan pembaruan status pengerjaan oleh teknisi lapangan.

4.6 File Page Aset (resources/js/Pages/Aset/Index.jsx)

Komponen React utama untuk menampilkan katalog aset medis:

4.7 File Layout (resources/js/Layouts/AuthenticatedLayout.jsx)

Menyediakan kerangka tampilan konsisten (sidebar dan header). Semua halaman dibungkus di dalam layout ini:

BAB V: PEMAHAMAN MENDALAM KONSEP TEKNOLOGI (LARAVEL, REACT, & REST API)
========================================================================

Bab ini berisi materi pemahaman fundamental secara konseptual. Dosen penguji sering kali menanyakan teori dasar di balik teknologi yang digunakan untuk menguji pemahaman mendalam Anda.

5.1 Laravel: Framework Backend PHP Modern
-----------------------------------------
Laravel adalah framework PHP MVC (Model-View-Controller) yang dirancang untuk mempermudah pengembangan aplikasi web yang aman, terstruktur, dan mudah dirawat. Konsep penting yang harus dipahami:
* Model-View-Controller (MVC): Pola arsitektur yang memisahkan data (Model), tampilan antarmuka (View), dan logika kontrol (Controller). Dalam arsitektur hybrid kita, React bertindak sebagai View, Laravel Model sebagai Model, dan Laravel Controller sebagai Controller.
* Eloquent ORM (Object-Relational Mapping): Teknik pemrograman untuk memetakan tabel database ke dalam bentuk objek bahasa pemrograman PHP. Kita tidak menulis query SQL manual (seperti SELECT * FROM aset), melainkan menggunakan sintaks objek seperti Aset::all() atau Aset::find(1).
* Middleware: Lapisan filter keamanan yang memeriksa setiap request HTTP sebelum masuk ke Controller. Contoh: Middleware auth memastikan pengguna sudah login session, dan middleware auth:sanctum memeriksa validitas token API Android.
* Database Transactions: Pengelompokan beberapa perintah database (insert/update/delete) menjadi satu kesatuan (unit of work). Jika salah satu perintah gagal, semua perintah dibatalkan secara otomatis (rollback) menggunakan DB::rollBack(). Jika semuanya sukses, data disimpan permanen menggunakan DB::commit(). Ini menjaga integritas data.

5.2 React.js: Library Frontend Berbasis Komponen
------------------------------------------------
React.js adalah library JavaScript yang dikembangkan oleh Meta untuk membangun antarmuka pengguna (UI) yang dinamis, cepat, dan interaktif. Konsep penting yang harus dipahami:
* Component-based Architecture: Aplikasi dipecah menjadi bagian-bagian kecil yang mandiri (reusable components). Contoh: Navbar, Sidebar, dan FormModal dibuat sekali dan dipanggil berulang-ulang di halaman yang berbeda.
* Virtual DOM (Document Object Model): React membuat salinan DOM asli browser di memori (Virtual DOM). Ketika data berubah, React membandingkan Virtual DOM dengan DOM asli dan hanya memperbarui elemen HTML yang berubah saja (Diffing Algorithm), bukan me-render ulang seluruh halaman. Ini membuat web berjalan sangat cepat.
* State vs Props: State adalah data internal yang dikelola dan diubah oleh komponen itu sendiri (lokal). Props adalah data eksternal yang dikirimkan oleh komponen induk ke komponen anak dan bersifat read-only (tidak bisa diubah langsung oleh komponen anak).
* React Hooks: Fungsi bawaan React untuk menggunakan fitur state dan lifecycle di function component. Contoh: useState() untuk membuat state lokal, dan useEffect() untuk menjalankan efek samping seperti mengambil data saat komponen pertama kali dirender.

5.3 REST API & Laravel Sanctum
------------------------------
REST (Representational State Transfer) API adalah standar arsitektur komunikasi data antara sistem yang berbeda (dalam proyek ini antara backend Laravel dengan klien mobile Android Kotlin) dengan menggunakan protokol HTTP. Konsep penting yang harus dipahami:
* Statelessness: Sifat utama REST API di mana server tidak menyimpan status session klien. Setiap HTTP request dari klien Android harus membawa informasi lengkap (token autentikasi) secara mandiri untuk diproses oleh server.
* HTTP Methods (Verbs): Standar komunikasi aksi: GET (mengambil data), POST (mengirim/membuat data baru), PUT/PATCH (memperbarui data), dan DELETE (menghapus data).
* Format Data JSON: Komunikasi data dilakukan menggunakan format JSON (JavaScript Object Notation), yaitu format teks ringan yang mudah dibaca oleh manusia dan mudah diparse oleh bahasa pemrograman apa pun (PHP di backend dan Kotlin di Android).
* Laravel Sanctum: Sistem autentikasi token bawaan Laravel yang sangat ringan. Sanctum menghasilkan string token acak (plainTextToken) yang disimpan di Android. Setiap request Android menyertakan token ini di header Authorization (Bearer Token) untuk diuji kecocokannya dengan token terenkripsi SHA-256 yang tersimpan di tabel database personal_access_tokens.

BAB VII: DEEP DIVE: KONSEP SESSIONS, TOKENS, DAN DATABASE MIGRATIONS
========================================================================

Tiga konsep ini adalah fondasi penting dalam pemrograman backend Laravel. Dosen penguji hampir pasti menanyakan perbedaan cara kerja autentikasi web (session) dengan Android (token), serta bagaimana migrasi bekerja.

7.1 Web Sessions (Autentikasi Stateful)
--------------------------------------
Session adalah mekanisme penyimpanan informasi sementara di sisi server untuk melacak status login pengguna di web browser. Cara kerjanya:
* Proses Autentikasi: Pengguna memasukkan email/password di browser -> Laravel mencocokkannya -> jika cocok, Laravel membuat file session unik di server (storage/framework/sessions/) dan mengirimkan Session ID ke browser dalam bentuk Cookie (bernama laravel_session).
* Pengiriman Cookie Otomatis: Web browser menyimpan cookie tersebut secara otomatis. Setiap kali pengguna berpindah halaman di web Anda, browser akan melampirkan cookie Session ID tersebut secara otomatis dalam request HTTP.
* Stateful: Disebut Stateful karena server secara aktif melacak dan menyimpan status login pengguna di memori/file server selama browser belum ditutup atau waktu session belum habis (expire).

7.2 API Tokens (Autentikasi Stateless)
--------------------------------------
Token adalah string acak unik yang digunakan untuk mengautentikasi pengguna pada REST API (mobile Android Kotlin). Cara kerjanya:
* Proses Autentikasi: Aplikasi Android mengirim email/password ke API login -> Laravel memverifikasi -> jika cocok, Laravel Sanctum membuat baris token acak baru di tabel database personal_access_tokens dan mengembalikan token string mentah tersebut ke aplikasi Android.
* Penyimpanan Manual: Aplikasi Android menyimpan token tersebut secara mandiri di memori lokal perangkat (SharedPreferences atau Datastore).
* Pengiriman Header Manual: Setiap kali Android mengirim request ke server (misal meminta data barang), Android harus melampirkan token tersebut secara manual di dalam header HTTP: Authorization: Bearer {token}.
* Stateless: Disebut Stateless karena server tidak menyimpan status session aktif untuk klien. Server hanya memeriksa apakah token yang dibawa di header HTTP cocok dengan yang ada di database. Jika cocok, data dikirim. Ini lebih hemat sumber daya server dan sangat fleksibel untuk integrasi multi-platform.
* Mengapa Android memakai Token bukan Session?: Karena aplikasi mobile (Android/iOS) tidak memiliki mekanisme bawaan untuk mengelola cookie session secara otomatis seperti web browser. Selain itu, token lebih aman dan kompatibel untuk komunikasi lintas platform berbasis API.

7.3 Database Migrations (Version Control untuk Database)
--------------------------------------------------------
Migrasi adalah fitur Laravel yang berfungsi sebagai sistem pengontrol versi (seperti Git) tetapi khusus untuk skema database. Dengan migrasi, struktur tabel didefinisikan menggunakan kode PHP, bukan query SQL manual.
* Manfaat Utama: Mempermudah kolaborasi tim developer atau proses penilaian oleh dosen. Cukup dengan menjalankan satu baris perintah di terminal baru, seluruh struktur database beserta relasi foreign key akan dibuat secara otomatis dengan tingkat akurasi 100% tanpa perlu ekspor-impor file SQL manual.
* Perintah Artisan Utama: Berikut adalah 3 perintah migrasi yang wajib dihafalkan fungsinya:
* 1. php artisan migrate: Menjalankan semua berkas migrasi baru yang belum pernah dijalankan sebelumnya di database.
* 2. php artisan migrate:rollback: Membatalkan atau memundurkan migrasi yang terakhir kali dijalankan (menghapus tabel/kolom yang baru dibuat). Ini berguna untuk membatalkan kesalahan skema.
* 3. php artisan migrate:fresh: Menghapus seluruh tabel di database (drop all tables) lalu menjalankan ulang seluruh berkas migrasi dari awal. Ini digunakan untuk mereset database ke keadaan kosong bersih.

BAB VIII: PEMAHAMAN STRUKTUR & DETAIL DATABASE (SCHEMAS & RELATIONSHIPS)
========================================================================

Dosen penguji hampir pasti menanyakan tentang struktur tabel, primary key, foreign key, dan hubungan relasional di dalam database MySQL projek MedTrack. Berikut adalah rincian fungsionalnya:

8.1 Penjelasan Detail 8 Tabel Inti MedTrack
-------------------------------------------
* 1. Tabel m_user (Data Pengguna / Staff):
  - Menyimpan data pengguna sistem yang terbagi menjadi role Admin_IPSRS, Staf_IPSRS, Staf_Logistik, dan Unit_RS. Menyimpan ID ruangan (id_ruang) sebagai foreign key yang merelasikan user ke unit kerjanya.
  - Primary Key: id_user
  - Foreign Key: id_ruang -> m_ruangan(id_ruang) untuk relasi penugasan ruangan staff.
* 2. Tabel m_ruangan (Data Unit Ruangan):
  - Menyimpan daftar unit kerja (IGD, ICU, OK, Poli, Logistik, IPSRS).
  - Primary Key: id_ruang
  - Unique Key: nama_ruang (untuk mencegah duplikasi nama ruangan).
* 3. Tabel t_aset (Data Alat Medis):
  - Menyimpan katalog inventaris alat medis beserta detail kelaikan operasional, nomor seri, dan koordinat tracking posisi GPS.
  - Primary Key: id_aset
  - Unique Key: kode_label (contoh: MED-001, MED-002)
  - Foreign Key: id_ruang_saat_ini -> m_ruangan(id_ruang) untuk melacak lokasi terkini alat.
  - Kolom Tracking: latitude, longitude (menyimpan koordinat GPS dari sensor mobile).
  - Kolom Kalibrasi: tgl_kalibrasi_terakhir, tgl_kalibrasi_berikutnya, tgl_kadaluarsa_sertif.
* 4. Tabel t_mutasi (Log Perpindahan Aset):
  - Mencatat riwayat pemindahan alat medis antar-ruangan untuk kebutuhan inventaris.
  - Primary Key: id_mutasi
  - Foreign Keys: id_aset -> t_aset(id_aset), id_ruang_asal -> m_ruangan(id_ruang), id_ruang_tujuan -> m_ruangan(id_ruang), user_id -> m_user(id_user).
  - Status: status_mutasi (Menunggu_Verifikasi, Disetujui, Ditolak).
* 5. Tabel t_troubleshoot (Tiket Laporan Kerusakan):
  - Menyimpan tiket aduan kerusakan alat medis (Work Order) yang diajukan oleh unit medis.
  - Primary Key: id_ticket
  - Foreign Keys: id_aset -> t_aset(id_aset), id_user_pelapor -> m_user(id_user), id_teknisi_penanggungjawab -> m_user(id_user).
  - Status: status_ticket (Open, Pengecekan, Dikerjakan, Menunggu Sign-Off, Closed).
* 6. Tabel t_troubleshoot_log (Riwayat Progress Tiket):
  - Mencatat kronologis perubahan status pengerjaan tiket kerusakan secara transaksional untuk audit trail log.
  - Primary Key: id_log
  - Foreign Keys: id_ticket -> t_troubleshoot(id_ticket), diubah_oleh -> m_user(id_user).
* 7. Tabel m_pemeliharaan & t_pemeliharaan_log (Penjadwalan PM):
  - Menyimpan jadwal pemeliharaan preventif (Preventative Maintenance) berkala untuk masing-masing aset.
  - Primary Key: id_pemeliharaan & id_log_pemeliharaan
  - Foreign Keys: id_aset -> t_aset(id_aset).
* 8. Tabel m_dokumen_mutu (Arsip Laporan & PDF):
  - Menyimpan berkas laporan bulanan mutu dan performa sarana prasarana yang diunggah dalam format PDF.
  - Primary Key: id_dokumen

8.2 Hubungan Relasional Antar-Tabel (Database Relationships)
------------------------------------------------------------
* One-to-Many (Ruangan ke Aset): Satu ruangan dapat memiliki banyak alat medis. Direlasikan melalui foreign key id_ruang_saat_ini di tabel t_aset.
* One-to-Many (Aset ke Troubleshoot): Satu alat medis dapat memiliki banyak catatan aduan kerusakan sepanjang masa pakainya. Direlasikan via foreign key id_aset di tabel t_troubleshoot.
* Many-to-One (Troubleshoot ke User): Banyak tiket kerusakan dapat dilaporkan oleh satu user pelapor (Unit_RS) atau ditangani oleh satu user teknisi (Staf_IPSRS).
* One-to-Many (Troubleshoot ke TroubleshootLog): Satu tiket kerusakan memiliki banyak log progres penanganan dari awal tiket dibuka hingga ditutup.


========================================================================
BAB IX: SIRKULASI ALIRAN DATA ANTARA BERKAS KODE (FILE) DAN TABEL DATABASE
========================================================================

Untuk memahami cara kerja framework secara utuh, berikut adalah sirkulasi/siklus aliran data yang menggambarkan bagaimana berkas kode program (View, Controller, Model) berinteraksi dengan database MySQL:

9.1 Sirkulasi Aliran Data pada Fitur Pelaporan Kerusakan (Work Order)
--------------------------------------------------------------------
* Frontend (View): Staf medis (Unit_RS) mengisi formulir laporan kerusakan di berkas resources/js/Pages/WorkOrder/Create.jsx, mengunggah foto bukti kerusakan, dan menekan tombol 'Kirim'. Request dikirim melalui Inertia.post('/workorder').
* Routing: Rute URL /workorder ditangkap oleh berkas routes/web.php dan diteruskan ke method store() pada berkas app/Http/Controllers/Web/WorkOrderController.php.
* Controller (Logika Backend): WorkOrderController memvalidasi data masukan (id_aset, deskripsi_kerusakan). Kemudian, Controller memanggil model Troubleshoot.php untuk menyimpan data.
* Model & Database: Berkas app/Models/Troubleshoot.php melakukan kueri INSERT ke tabel t_troubleshoot di MySQL untuk mencatat baris data baru dengan status_ticket = 'Open'.
* Umpan Balik (Response): Controller mengembalikan rute redirect kembali ke halaman index Work Order dengan menyertakan flash message sukses. Berkas resources/js/Pages/WorkOrder/Index.jsx menerima shared props, me-reload tabel secara reaktif, dan menampilkan tiket baru tersebut di daftar.

9.2 Sirkulasi Aliran Data pada Fitur Pengajuan Mutasi Aset
---------------------------------------------------------
* Frontend (View): Pengguna Unit_RS membuka berkas resources/js/Pages/Mutasi/Create.jsx, memilih alat medis yang ingin dimutasi, memilih unit ruangan tujuan, dan mengklik tombol 'Ajukan Mutasi'.
* Routing & Controller: Request POST dikirim ke rute /mutasi dan diolah oleh MutasiController@store. Berkas ini berinteraksi dengan model Mutasi.php untuk membuat record log baru dengan status 'Menunggu_Verifikasi' pada tabel t_mutasi.
* IPSRS Verifikasi (Approval): Admin IPSRS membuka halaman resources/js/Pages/Mutasi/Index.jsx, melihat status pending, dan menekan tombol 'Approve'.
* Database Transaction: Request persetujuan dikirim ke MutasiController@verify. Di dalam blok DB::transaction():
  1. Model Aset.php diinstruksikan untuk memperbarui kolom id_ruang_saat_ini pada tabel t_aset (mengubah lokasi fisik alat medis).
  2. Model Mutasi.php memperbarui status_mutasi di tabel t_mutasi menjadi 'Disetujui'.
* Selesai & Render: DB::commit() menyimpan perubahan secara permanen, dan Inertia memicu re-render halaman sehingga status di layar berubah menjadi Approved dan lokasi aset terbarui secara real-time.

BAB X: BEDAH DETAIL KONSEPTUAL DAN PROTOKOL KUNCI FRAMEWORK
========================================================================

10.1 Protokol Komunikasi Inertia.js (Bridges Between Backend and Frontend)
--------------------------------------------------------------------------
Inertia.js menghilangkan kebutuhan pembuatan endpoint API JSON manual untuk klien web. Protokol ini bekerja dengan mengirimkan header request khusus:
* X-Inertia: true -> Header HTTP yang dikirimkan oleh browser untuk memberi tahu backend Laravel bahwa ini adalah request dinamis dari React, bukan request halaman biasa.
* Jika header X-Inertia bernilai true, Laravel melalui middleware Inertia akan membatalkan rendering file view Blade HTML mentah, dan sebaliknya mengembalikan representasi payload JSON murni yang berisi nama komponen Page React dan array props data.
* Jika pengguna memasukkan URL secara manual di browser (request biasa pertama kali), Laravel akan menyajikan halaman Blade utama (app.blade.php) yang memuat file bundler JavaScript (vite/app.jsx), lalu Inertia akan me-mount halaman React yang sesuai di dalam elemen div root.

10.2 Keamanan Otorisasi Berbasis Peran (Role-Based Access Control / RBAC) di Laravel
------------------------------------------------------------------------------------
Sistem keamanan MedTrack memadukan pengamanan berlapis:
* Middleware 'auth' Session -> Memastikan pengguna sudah login di level web browser sebelum rute routes/web.php diproses.
* Logic-Level Authorization -> Di dalam controller, hak akses CRUD diuji ketat menggunakan fungsi abort(403) jika user tidak memiliki role yang diizinkan, sehingga mencegah eksploitasi URL (seperti memanggil edit aset menggunakan URL POST manual).

10.3 State Synchronization dan React Reactivity
-----------------------------------------------
Saat form disubmit menggunakan Inertia (misalnya router.post('/aset', data)), Inertia secara otomatis mengelola siklus state React:
* Inertia mengirim request POST -> Laravel memproses dan melakukan redirect kembali -> Inertia di sisi frontend menangkap redirect tersebut, memperbarui properti props komponen secara dinamis di memori, dan memicu Virtual DOM React untuk me-render ulang elemen-elemen UI yang berubah saja tanpa melakukan full page refresh.
