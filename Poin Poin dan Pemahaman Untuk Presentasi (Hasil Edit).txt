🏥 PANDUAN DETAIL PEMAHAMAN TEKNIS PROJEK AKHIR (MEDTRACK)
Materi Pendalaman Struktur Kode, Alur Data, Otorisasi Peran, dan Integrasi Laravel + React Inertia Khusus untuk Menjawab Pertanyaan Dosen Penguji secara Rinci

========================================================================
BAB I: BEDAH STRUKTUR DIREKTORI PROYEK (LARAVEL + REACT INERTIA)
========================================================================

Dosen penguji sering kali menanyakan bagaimana backend dan frontend terhubung secara arsitektural. Proyek ini menggunakan arsitektur Hybrid Monolith dengan Inertia.js sebagai jembatan dinamis. Berikut adalah penjelasan struktur folder secara fungsional:

Sisi Backend (Laravel 10)
-------------------------
* app/Http/Controllers/Web/
  - Folder ini menyimpan seluruh logika bisnis sistem. Controller bertugas mengambil data dari Model dan mengembalikannya ke frontend melalui Inertia::render.
* app/Http/Controllers/Api/
  - Berisi berkas-berkas kontroler API untuk melayani pertukaran data koordinat GPS pelacak lokasi aset medis dari aplikasi mobile Android dalam bentuk JSON.
* app/Models/
  - Folder representasi tabel database. Menggunakan ORM Eloquent sehingga kueri database direpresentasikan sebagai objek PHP. Di dalamnya terdapat file Aset.php, User.php, Ruangan.php, Troubleshoot.php (Work Order), TroubleshootLog.php, Mutasi.php, Pemeliharaan.php, PemeliharaanLog.php, dan DokumenMutu.php.
* database/migrations/
  - Menyimpan skema pembentukan tabel database beserta constraint relasi foreign key.
* database/seeders/
  - Menyimpan seeder data bawaan untuk keperluan demonstrasi akun pengujian default.
* routes/web.php
  - Tempat mendaftarkan rute akses web client yang dilindungi middleware 'auth' session.
* routes/api.php
  - Tempat mendaftarkan rute REST API untuk Android yang diamankan oleh middleware token 'auth:sanctum'.

Sisi Frontend (React.js + Tailwind CSS)
---------------------------------------
* resources/js/Pages/
  - Menyimpan file komponen halaman web. Setiap file React JSX di sini (seperti Dashboard.jsx, Aset/Index.jsx, WorkOrder/Index.jsx, Mutasi/Index.jsx) merender antarmuka pengguna halaman web secara dinamis.
* resources/js/Layouts/AuthenticatedLayout.jsx
  - Kerangka tata letak halaman dashboard (sidebar, header notifikasi, scanner QR code).
* resources/js/Components/
  - Menyimpan komponen UI kecil reusable (seperti Modal, Dropdown, Button).

BAB II: PENJELASAN DETIL SETIAP FILE KODE PROGRAM
========================================================================

Di bawah ini adalah rincian detail fungsi dan metode kunci untuk setiap file kode program di dalam proyek MedTrack:

1. Model Eloquent (app/Models/)
-------------------------------
* User.php
  - Memetakan tabel m_user. Menyimpan informasi profil staff, peran (role), password, dan relasi ke unit/ruangan kerja.
  - Relasi ruangan(): belongsTo(Ruangan::class, 'id_ruang') untuk mendapatkan unit kerja user.
* Aset.php
  - Memetakan tabel t_aset. Menyimpan data inventaris alat medis, serial number, status operasional, koordinat lokasi GPS tracking, tanggal kalibrasi, dan file QR Code.
  - Relasi ruangan(): belongsTo(Ruangan::class, 'id_ruang_saat_ini') untuk melacak lokasi ruang terkini alat.
  - Relasi troubleshoot(): hasMany(Troubleshoot::class, 'id_aset') untuk mendapatkan riwayat kerusakan alat.
* Ruangan.php
  - Memetakan tabel m_ruangan. Menyimpan data ruangan di rumah sakit.
  - Relasi asets(): hasMany(Aset::class, 'id_ruang_saat_ini') untuk mendapatkan semua inventaris di ruangan tersebut.
* Mutasi.php
  - Memetakan tabel t_mutasi. Menyimpan riwayat pengajuan pemindahan alat.
  - Relasi aset(): belongsTo(Aset::class, 'id_aset')
  - Relasi ruanganAsal(): belongsTo(Ruangan::class, 'id_ruang_asal')
  - Relasi ruanganTujuan(): belongsTo(Ruangan::class, 'id_ruang_tujuan')
* Troubleshoot.php
  - Memetakan tabel t_troubleshoot. Menyimpan tiket laporan kerusakan alat medis.
  - Relasi pelapor(): belongsTo(User.class, 'id_user_pelapor')
  - Relasi teknisi(): belongsTo(User.class, 'id_teknisi_penanggungjawab')
  - Relasi aset(): belongsTo(Aset::class, 'id_aset')
  - Relasi ticketLogs(): hasMany(TroubleshootLog::class, 'id_ticket')

2. Web Controllers (app/Http/Controllers/Web/)
----------------------------------------------
* DashboardController.php
  - Menyajikan dasbor overview. Menghitung ringkasan statistik (jumlah aset, total kerusakan, mutasi pending) dan memuat agenda kalibrasi harian.
  - index(): Mengambil data statistik kartu ringkasan dan kalender pemeliharaan.
  - reschedule(): Logika pembaruan tanggal kalibrasi alat secara interaktif dari kalender sidebar.
* AsetController.php
  - Menangani manajemen inventaris alat medis.
  - index(): Menampilkan katalog aset. Menyaring data otomatis per unit untuk Unit_RS, dan menonaktifkan filter jika ada parameter pencarian (scan QR).
  - checkCanCreate() & checkCanEditOrDelete(): Membatasi hak akses CRUD aset agar tidak dapat diakses oleh Unit_RS, Staf_Logistik (hanya bisa create), serta teknisi budi_ipsrs, hendra_ipsrs, dan agus_ipsrs.
  - store() & update(): Melakukan validasi pengisian data aset medis dan meng-generate file gambar QR Code secara dinamis.
* WorkOrderController.php
  - Menangani alur pelaporan kerusakan alat medis.
  - store(): Menyimpan tiket kerusakan baru dari Unit_RS dengan status default 'Open'.
  - assignTeknisi(): Meng-assign teknisi penanggungjawab (budi_ipsrs, hendra_ipsrs, atau agus_ipsrs) ke dalam tiket. Diblokir jika diakses oleh teknisi lapangan itu sendiri.
  - updateStatus(): Mengubah status progres perbaikan alat (Open -> Pengecekan -> Dikerjakan -> Menunggu Sign-Off) oleh teknisi penanggungjawab.
* MutasiController.php
  - Mengelola alur pemindahan/mutasi alat medis antar ruangan.
  - store(): Mengajukan pemindahan alat medis dengan status 'Menunggu_Verifikasi'.
  - verify(): Logika verifikasi persetujuan (Approve/Reject) mutasi oleh IPSRS yang menggunakan DB::transaction() untuk memperbarui kolom id_ruang_saat_ini pada aset sekaligus mencatat log mutasi.
* DirektoriController.php
  - Mengelola master data ruangan rumah sakit dan direktori staff/SDM.
  - checkAccess(): Memproteksi CRUD ruangan dan profil staff agar menolak role Unit_RS, Staf_Logistik, dan teknisi lapangan (Budi/Hendra/Agus).

3. API Controllers (app/Http/Controllers/Api/)
----------------------------------------------
* ApiAuthController.php
  - Melayani proses autentikasi (login/logout) aplikasi mobile tracker Android Kotlin dan menghasilkan string plainTextToken Sanctum.
* ApiAsetController.php
  - Melayani kueri data inventaris aset dan menerima update data posisi koordinat GPS tracker dari aplikasi Android untuk disimpan ke tabel t_aset.

4. Halaman React Pages (resources/js/Pages/)
--------------------------------------------
* Dashboard.jsx
  - Merender visual statistik dasbor, chart grafik tingkat kerusakan, dan penampakan daftar notifikasi terkini.
* Aset/Index.jsx
  - Katalog data aset medis lengkap dengan kartu informasi, tombol tambah/edit/hapus (sesuai otorisasi), dan integrasi scan QR Code.
* Aset/Show.jsx
  - Detail spesifikasi aset medis dan visualisasi peta pelacakan posisi GPS real-time.
* WorkOrder/Index.jsx
  - Tiket pelaporan kerusakan, dilengkapi dropdown inline untuk mengubah status perbaikan (bagi teknisi) dan assign teknisi (bagi admin).
* Mutasi/Index.jsx
  - Daftar riwayat mutasi alat dilengkapi pill badge status verifikasi (Pending, Approved, Rejected).
* Direktori/Index.jsx
  - Grid data ruangan dan direktori staff rumah sakit.
* Layouts/AuthenticatedLayout.jsx
  - Layout utama pembungkus halaman. Di sini diletakkan menu navigasi sidebar, bell notifikasi dengan filter role, dan modal kamera scanner QR Code.

BAB III: ALUR DATA DAN REQUEST-RESPONSE LIFECYCLE (BAGAIMANA APLIKASI BEKERJA)
========================================================================

Siklus Request Web (Menggunakan Protokol Inertia.js)
----------------------------------------------------
Inertia bertindak sebagai jembatan. Alur kerja ketika pengguna mengakses halaman Aset Medis adalah sebagai berikut:
1. Pengguna mengklik menu 'Aset Medis' di Sidebar.
2. Browser mengirimkan request internal (Inertia request XHR/AJAX) ke rute /aset.
3. Laravel mencocokkan rute di web.php dan memanggil AsetController@index.
4. AsetController melakukan query database melalui Eloquent: Aset::with('ruangan')->get.
5. Controller mengembalikan respon: return Inertia::render('Aset/Index', ['asets' => $asets]);
6. Inertia menangkap kembalian tersebut, lalu mengirimkan payload berupa data JSON (bukan HTML mentah) ke frontend.
7. Komponen React di resources/js/Pages/Aset/Index.jsx menerima data tersebut melalui props (asets) lalu merendernya secara dinamis di browser tanpa refresh halaman penuh.

Siklus Request API Mobile (Laravel Sanctum)
-------------------------------------------
Alur kerja ketika aplikasi Android pelacak lokasi mengirim data ke server:
1. Aplikasi Android mengirim HTTP POST request ke http://alamat-server/api/tracking.
2. Android menyertakan header Authorization: Bearer {token} yang diperoleh saat login.
3. Laravel mencocokkan rute di routes/api.php dan memprosesnya melalui middleware auth:sanctum.
4. Middleware memverifikasi keaslian token pada tabel personal_access_tokens di database.
5. Jika valid, request diteruskan ke Api\AsetController@updateTracking yang menyimpan koordinat GPS terbaru ke database.


========================================================================
BAB IV: BEDAH LOGIKA DAN KODE CRITICAL
========================================================================

Notifikasi Lonceng Dinamis di Layout Header
-------------------------------------------
Notifikasi dikumpulkan secara terpusat di HandleInertiaRequests.php berdasarkan role pengguna yang login:
* IPSRS: Laporan kerusakan baru (Open WO), mutasi menunggu verifikasi, dan jadwal pemeliharaan (PM).
* Logistik: Aset medis yang tanggal sertifikat kalibrasinya sudah kadaluarsa (expired).
* Unit_RS: Status penanganan tiket kerusakan aktif di unit ruangan mereka.
* Fungsi Dismiss: Di frontend AuthenticatedLayout.jsx, saat notifikasi diklik, ID item disimpan ke localStorage. Notifikasi tersebut langsung hilang dari lonceng secara real-time dan browser mengarahkan pengguna ke halaman form tujuan.

Keamanan Mutasi Aset (DB Transactions)
--------------------------------------
Proses mutasi di MutasiController dibungkus dalam blok DB::transaction() untuk menjaga konsistensi. Jika salah satu proses gagal, seluruh kueri dibatalkan (Rollback) agar tidak ada data ruangan aset yang tidak sinkron.

BAB V: PEMAHAMAN MENDALAM KONSEP TEKNOLOGI (LARAVEL, REACT, & REST API)
========================================================================

Laravel (MVC & ORM)
-------------------
* MVC: Pemisahan data (Model), antarmuka (View/React), dan logika (Controller) agar kode terstruktur.
* Eloquent ORM: Mengubah baris tabel database menjadi objek PHP (Aset::find(1)).
* Middleware: Filter keamanan HTTP request (seperti mengecek login session atau token API).

React.js (Virtual DOM & State)
------------------------------
* Virtual DOM: React membandingkan salinan DOM di memori dengan DOM asli dan hanya me-render bagian HTML yang berubah saja (Diffing Algorithm), sehingga sangat cepat.
* State vs Props: State adalah variabel lokal internal komponen (dapat diubah), sedangkan Props adalah data kiriman dari parent/controller (read-only).

========================================================================
BAB VII: DEEP DIVE: KONSEP SESSIONS, TOKENS, DAN DATABASE MIGRATIONS
========================================================================

* Web Sessions (Stateful): Browser menyimpan Session ID di Cookie (laravel_session) yang dilampirkan otomatis pada setiap request HTTP. Server melacak status login di memori.
* API Tokens (Stateless): Android menyimpan token secara mandiri dan mengirimkannya di header Authorization. Server hanya memeriksa apakah token valid tanpa menyimpan status session.
* Database Migrations: Version control untuk skema database. php artisan migrate digunakan untuk memperbarui tabel database secara otomatis tanpa impor file SQL manual.
